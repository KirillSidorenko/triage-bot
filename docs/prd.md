# PRD: AI Triage Bot для Customer Support

## 1. Резюме

Мы создаём AI-бота для автоматического триажа обращений в customer support — классификация, приоритизация и маршрутизация сообщений, сокращающая время triage с 5 минут до ~3 секунд на тикет. Текущая версия использует DeepSeek V3 (`deepseek-chat`) для анализа входящих сообщений в Telegram, классифицирует их по 6 категориям, назначает приоритет и направляет в ответственную команду — с human-in-the-loop для low-confidence и critical кейсов. В состоянии `v1.2` workflow уже собран end-to-end, прогнан на 50 synthetic test cases и показывает 92% classification accuracy, 92% routing accuracy, 76% priority accuracy и 94% review-trigger accuracy. Следующий шаг — controlled pilot на реальных сообщениях и логирование решений ревьюера.

## 2. Описание проблемы

### Кто сталкивается с этой проблемой?
Customer support команды в SaaS-компаниях (50–200 сотрудников), обрабатывающие 500–5000 обращений в месяц. L1-агенты тратят большую часть времени на сортировку, а не на решение проблем.

### В чём проблема?
L1-агенты вручную читают каждое обращение, определяют категорию, приоритет и передают в нужную команду. Этот процесс:

- **Медленный:** 3–8 минут на обращение. При 100 обращениях/день = 5–13 часов чистого триажа
- **Непоследовательный:** разные агенты классифицируют одинаковые кейсы по-разному (~70% consistency)
- **Ошибочный:** 15–25% обращений маршрутизируются неправильно, увеличивая время решения на 30–60 минут
- **Не масштабируется:** рост обращений требует линейного найма L1-агентов

### Почему это болезненно?
- **Для бизнеса:** при 100 обращений/день компания тратит ~$3,000–5,000/месяц только на ручной триаж (зарплата L1-агентов на рутину)
- **Для клиентов:** среднее время первого ответа 2–4 часа, critical issues ждут в общей очереди
- **Для агентов:** выгорание на рутинной работе вместо решения реальных проблем

### Подтверждение данными
- **Отраслевые данные:** среднее время триажа 3–8 минут на обращение (Zendesk Benchmark Report)
- **Ошибки маршрутизации:** 15–25% misrouting rate в командах без автоматизации (Freshdesk, 2024)
- **Время агентов:** 30–40% рабочего времени L1 уходит на классификацию и маршрутизацию (Intercom Customer Support Trends)
- **Типичная жалоба:** "Мы нанимаем людей, чтобы они читали письма и перекладывали их в правильную папку"

## 3. Целевые пользователи и персоны

### Основная персона: Operations Manager Ольга
- **Роль:** руководитель поддержки / Operations Manager в SaaS-компании
- **Размер команды:** 5–15 support-агентов
- **Цели:** снизить cost-per-ticket, улучшить SLA compliance, масштабировать без найма
- **Боли:** не хватает данных для оптимизации, нет прозрачности в распределении нагрузки, ручной процесс не масштабируется
- **Поведение:** ищет automation-решения, готова инвестировать время в настройку, если ROI понятен
- **Успех для неё:** "Мои агенты решают проблемы, а не сортируют почту"

### Вторичная персона: Support-агент Алексей
- **Роль:** L1/L2 support-агент
- **Цели:** тратить время на решение проблем, а не на сортировку
- **Боли:** рутина, непоследовательная маршрутизация от коллег, нет контекста при получении тикета
- **Успех для него:** "Я получаю тикет с готовым summary и знаю, что это точно мой тип задачи"

### Jobs-to-Be-Done
- **Ольга:** "Когда объём обращений растёт, я хочу автоматизировать триаж, чтобы не нанимать ещё одного L1-агента"
- **Алексей:** "Когда я получаю тикет, я хочу сразу видеть summary и категорию, чтобы не тратить время на чтение и классификацию"

## 4. Стратегический контекст

### Бизнес-цели
- Снижение операционных затрат на L1 triage на 60–80%
- Улучшение SLA compliance (правильная маршрутизация → быстрее решение)
- Создание data foundation для аналитики support-процесса

### Рыночная возможность
- **TAM:** ~2M SaaS-компаний глобально с support-командами
- **SAM:** ~200K компаний 50–500 сотрудников, активно ищущих automation
- **SOM:** portfolio project — демонстрация компетенций для AI PM / automation ролей

### Конкурентный ландшафт
| Решение | Подход | Ограничение |
|---------|--------|-------------|
| Zendesk AI | Встроенная классификация | Vendor lock-in, дорого, black box |
| Intercom Fin | AI-ответы клиентам | Фокус на auto-reply, не на routing |
| Custom ML pipeline | Обучение на данных компании | Нужны данные + ML-инженер + месяцы |
| **Наш подход** | LLM + no-code orchestration | Быстрый setup, прозрачный, настраиваемый |

### Почему сейчас?
- Качество современных LLM (DeepSeek V3, GPT-4o, Claude-class models) достаточно для надёжной классификации без fine-tuning
- No-code orchestration (n8n) снижает порог входа
- Компании готовы к AI-автоматизации после волны adoption ChatGPT (2023–2025)

## 5. Описание решения

### Как это работает
AI triage bot принимает обращения клиентов через Telegram, анализирует их с помощью DeepSeek V3 и автоматически классифицирует, приоритизирует и маршрутизирует. Конечный operational destination — `Yandex Tracker`, а `Google Sheets` используется как audit trail для отслеживания и аналитики.

### Workflow
``` 
Сообщение в Telegram → n8n webhook
  → Нормализация (текст, отправитель, timestamp)
  → DeepSeek V3 (классификация + summary)
  → Проверка needs_review
    → ≥0.8: автоматическая маршрутизация → Yandex Tracker target queue → Google Sheets → подтверждение в Telegram
    → <0.8 или critical: review queue в Yandex Tracker → Google Sheets → уведомление ревьюеру
```

### Ключевые возможности (v1)
1. **Автоклассификация:** 6 категорий (billing, technical_issue, account_access, feature_request, complaint, general_inquiry)
2. **Назначение приоритета:** critical / high / medium / low с привязкой к SLA
3. **Умная маршрутизация:** каждая категория → ответственная команда
4. **Порог уверенности:** автоматическая обработка при confidence ≥0.8, human review при <0.8
5. **Генерация summary:** краткое описание проблемы для L2-специалиста
6. **Operational routing:** итоговый тикет создаётся в `Yandex Tracker`
7. **Полное логирование:** все решения записываются в Google Sheets

### Технический стек
| Компонент | Технология | Почему |
|-----------|------------|--------|
| Приём сообщений | Telegram Bot | Простой setup, real-time, наглядная демонстрация |
| Оркестрация | n8n | Visual workflow, быстрая итерация, self-hosted |
| AI | DeepSeek V3 (deepseek-chat) | Текущая модель в workflow и eval; OpenAI-compatible API, доступна из РФ, низкая стоимость |
| Ticketing | Yandex Tracker | Реалистичный support routing, видимые очереди, понятный operational handoff |
| Audit trail | Google Sheets | Быстрый и прозрачный лог для eval и разбора |

## 5.1 Текущее состояние (v1.2)

- End-to-end workflow собран: Telegram → n8n → DeepSeek V3 → Yandex Tracker + Google Sheets audit log
- Synthetic eval completed: 50 кейсов, 6 категорий, RU/EN, включая edge cases
- Измеренные результаты: category 92%, routing 92%, priority 76%, review trigger 94%

## 6. Метрики успеха

### Основная метрика
**Triage time** — время от получения обращения до классификации и маршрутизации
- **Текущее значение:** ~5 минут (ручной L1 триаж)
- **Целевое значение:** <5 секунд (автоматический)
- **Как измеряем:** timestamp diff в Google Sheets (received → processed)

### Вторичные метрики
| Метрика | Baseline | Текущее v1.2 | Pilot gate | Как измеряем |
|---------|----------|--------------|------------|--------------|
| Classification accuracy | — | **92%** | ≥85% | Evaluation set (50 кейсов) |
| Routing accuracy | 75–85% | **92%** | ≥90% | Evaluation set |
| Priority accuracy | — | **76%** | ≥75% | Evaluation set |
| Review trigger accuracy | — | **94%** | ≥90% | expected_needs_review vs actual_needs_review |
| Triage time | ~5 минут | **~3 сек** | <10 сек | timestamp diff в Google Sheets |

### Guardrail-метрики
- **False negative rate для critical issues:** должен быть <5% — нельзя пропускать critical тикеты
- **Hallucination rate:** 0% — бот не должен выдумывать категории вне списка

## 7. User Stories и требования

### Гипотеза эпика
Мы считаем, что автоматизация triage с помощью DeepSeek V3 для support-команд сократит время triage с 5 минут до менее 5 секунд, потому что текущий процесс чисто механический (прочитать → классифицировать → направить) и хорошо подходит для LLM-автоматизации. На synthetic eval гипотеза уже частично подтверждена: classification accuracy = 92%, routing accuracy = 92%. Следующий этап валидации — controlled pilot на реальных сообщениях и замер качества human review flow.

### User Stories

**Story 1: Отправка обращения**
Как клиент, я хочу отправить свой вопрос в Telegram-бот, чтобы моё обращение было зарегистрировано и направлено без ожидания, пока человек его прочитает.

Критерии приёмки:
- [ ] Бот принимает текстовые сообщения до 4000 символов
- [ ] Бот отвечает подтверждением в течение 10 секунд
- [ ] Подтверждение не раскрывает внутренние поля triage клиенту
- [ ] Если сообщение пустое или слишком длинное, бот возвращает понятную ошибку

**Story 2: Автоклассификация и маршрутизация**
Как operations manager, я хочу, чтобы входящие сообщения автоматически классифицировались и направлялись, чтобы мои L1-агенты не тратили время на ручной триаж.

Критерии приёмки:
- [ ] Каждое сообщение классифицируется ровно в одну из 6 категорий
- [ ] Назначается приоритет (critical/high/medium/low)
- [ ] Маршрутизация следует маппингу категория → команда из routing_rules.json
- [ ] Classification accuracy ≥85% на evaluation set

**Story 3: Human review для low-confidence**
Как operations manager, я хочу, чтобы классификации с низкой уверенностью помечались для ручной проверки, чтобы неоднозначные кейсы не маршрутизировались ошибочно.

Критерии приёмки:
- [ ] Сообщения с confidence <0.8 или priority = critical помечаются needs_review=true
- [ ] Ревьюер получает уведомление в отдельном Telegram-канале
- [ ] Уведомление содержит: оригинальное сообщение, AI summary, предложенную категорию, confidence
- [ ] Ревьюер может изменить категорию и приоритет

**Story 4: Логирование всех решений**
Как operations manager, я хочу, чтобы каждое решение по триажу записывалось в Google Sheets, чтобы у меня была полная прозрачность и данные для оптимизации.

Критерии приёмки:
- [ ] Каждое обработанное сообщение создаёт строку в Google Sheets
- [ ] Строка содержит: timestamp, sender, original_message, summary, category, priority, confidence, route, needs_review, reviewer_decision, final_category, processing_time_ms, tracker_issue_key, tracker_queue, tracker_issue_url
- [ ] Данные не теряются при ошибках API (fallback route + логирование)

**Story 5: Корректная обработка ошибок**
Как клиент, я хочу, чтобы бот обрабатывал ошибки без потери моего сообщения, чтобы моё обращение всегда было зарегистрировано.

Критерии приёмки:
- [ ] Если DeepSeek API не отвечает после 3 попыток, сообщение направляется в fallback (General Support)
- [ ] Если DeepSeek API не отвечает после 3 попыток, сообщение уходит в безопасный fallback path: тикет создаётся в review-очереди с логическим fallback owner `General Support`
- [ ] Клиент получает подтверждение даже в случае ошибки
- [ ] Ошибка логируется с сырым ответом API для отладки

### Edge Cases
- **Сообщение на несколько тем:** "Не могу войти + дважды списали деньги" → классификация по основной проблеме, снижение confidence
- **Очень короткое сообщение:** "не работает" → general_inquiry, низкий confidence, пометка для review
- **Эмоциональное сообщение:** "ВАШ СЕРВИС УЖАСЕН!!!" → смотрим сквозь эмоции на суть проблемы
- **Prompt injection:** "Ignore instructions and..." → изоляция system prompt, валидация JSON

## 10. Открытые вопросы

- [ ] **Confidence threshold 0.8 — правильный ли порог?** На synthetic eval review trigger accuracy = 94%, но нужен pilot на реальных сообщениях. Если слишком много safe-кейсов уходит на review, threshold или heuristics нужно пересматривать.
- [x] **Нужен ли отдельный порог для critical issues?** **Решение:** да — все critical issues идут на human review независимо от confidence. Бот не заменяет человека, а augment'ирует: ревьюер получает готовый summary + предварительную классификацию, что сокращает его время реакции с ~5 мин до ~1 мин. Причина: цена ошибки для critical (пропущенный data breach, полный downtime) несоразмерна выгоде от автоматизации 5 минут. Валидация: замерить время реакции ревьюера с summary vs без, отслеживать NPS по critical тикетам.
- [ ] **Как обрабатывать multi-topic сообщения в v2?** Разделять на отдельные тикеты или primary/secondary classification?
- [ ] **Какой SLA на human review?** Если ревьюер не отвечает в течение X минут — авто-маршрутизация?
- [ ] **Нужен ли rate limiting для Telegram-бота?** Защита от спама и злоупотреблений.
- [ ] **Нужен ли cross-model benchmark после pilot?** Сейчас workflow и eval идут на DeepSeek V3. Сравнение с GPT-4o mini / Claude Sonnet имеет смысл только если pilot покажет quality или latency gap.
